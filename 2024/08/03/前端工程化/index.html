Webpack

是什么：

webpack 是一个用于现代JavaScript应用程序的静态模块打包工具

模块整合能力（.js -> bundle）/万物皆可模块/编译代码(如ES6 -> ES5)

构建流程

  1. 初始化：从配置文件（webpack.config.js）和shell中读取参数，初始化compiler编译对象
  2. 编译构建： compile(开始编译，构建Compilation对象) -> make编译模块（从entry出发，调用loader编译模块，解析输出抽象语法树，再找出依赖的模块递归进行） -> build module(完成模块编译)
  3. 输出： seal输出资源（生成chunks） -> emit输出完成（通过配置的出口）

loader

loader 用于对模块的”源代码”进行转换，在 import 或”加载”模块时预处理文件
配置：

module.exports = {
  module: {
    rules: [
      {
        test: /\.css$/,
        use: [
          { loader: 'style-loader' },
          {
            loader: 'css-loader',
            options: {
              modules: true
            }
          },
          { loader: 'sass-loader' }//解析：从后往前
        ]
      }
    ]
  }
};

常见: style- css-（一起用） less/sass- postcss- file/url-(静态资源) babel-(es6 to es5)
特性：同步/异步，链式调用
编写：

// 导出一个函数，source为webpack传递给loader的文件源内容
module.exports = function(source) {
    const content = doSomeThing2JsString(source);
    
    // 如果 loader 配置了 options 对象，那么this.query将指向 options
    const options = this.query;
    
    // 可以用作解析其他模块路径的上下文
    console.log('this.context');
    
    /*
     * this.callback 参数：
     * error：Error | null，当 loader 出错时向外抛出一个 error
     * content：String | Buffer，经过 loader 编译后需要导出的内容
     * sourceMap：为方便调试生成的编译后内容的 source map
     * ast：本次编译生成的 AST 静态语法树，之后执行的 loader 可以直接使用这个 AST，进而省去重复生成 AST 的过程
     */
    this.callback(null, content); // 异步
    return content; // 同步
}

plugin

插件
配置：

module.exports = {
  ...
  plugins: [
    new webpack.ProgressPlugin(),
    new HtmlWebpackPlugin({ template: './src/index.html' }),
  ],
};

本质是具有apply方法的js对象
常见：

  • HtmlWebpackPlugin 自动生成html，引入js模块
  • clean-webpack-plugin 清理构建目录
  • mini-css-extract-plugin 提取css到单独文件
  • copy-webpack-plugin 复制文件或目录（如public）到执行区域
    编写：
class MyPlugin {
    // Webpack 会调用 MyPlugin 实例的 apply 方法给插件实例传入 compiler 对象
  apply (compiler) {
    // 找到合适的事件钩子，实现自己的插件功能
    compiler.hooks.emit.tap('MyPlugin', compilation => {
        // compilation: 当前打包构建流程的上下文
        console.log(compilation);
        
        // do something...
    })
  }
}

与loader区别：
概念上； 运行时机：loader在打包前，plugin在整个编译周期

热更新(HMR)

开启： devServer: {hot: true}
原理：

当文件变化时，webpack监听到并重新编译打包，生成唯一的hash值 -> devServer向浏览器发送改动信息（包含hash值）（两者通过webSocket连接）
->浏览器获取模块变化，触发render
两个服务端： express server 静态资源服务（初次构建时） socket server (通信服务)

proxy

使用dev-server配置：

module.exports = {
    // ...
    devServer: {
        contentBase: path.join(__dirname, 'dist'),
        compress: true,
        port: 9000,
        proxy: {
            '/api': {
                target: 'https://api.github.com'//目标地址
                pathRewrite: ...//重写路径，如删除/api
                changeOrigin: ...//是否更新代理后请求的 headers 中host地址
            }
        }
        // ...
    }
}

工作原理： 使用代理服务器解决跨域问题

优化

  1. 性能
  • 压缩JS/CSS/HTML/文件/图片
    terser/css-minimizer-webpack-plugin/minify/compression-webpack-plugin
  • tree-shaking (usedExports/sideEffects)
  • 代码分离，按需加载(splitChunks)
  • 内联chunk
  1. 构建速度
  • loader 配置include/exclude
  • resolve.extensions(自动添加扩展名)
  • resolve.modules(从哪里寻找第三方模块。如node_modules)
  • resolve.alias(别名，如@ -> ./src)
  • cache-loader(缓存打包结果)
  • terser启动多线程
  • sourceMap(开发环境和生产环境分开)

Vite

vite开发环境依赖esbuild进行预构建，生产环境则依赖rollup进行打包
vite支持原生ESMoudle，使用esbuild预构建，先启动开发服务器再按需编译，效率高于wp，提高冷启动速度
vite利用了http2并发请求的优势加速
vite监听到文件变更后，会用websocket通知浏览器，重新发起新的请求，只对该模块进行重新编译，然后进行替换，而wp会对整个项目重新进行打包（使用缓存机制解决）
生产环境下，vite使用rollup进行打包（rollup优势：体积小，支持tree-shaking，适用于js库的打包）
