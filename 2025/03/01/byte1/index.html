微前端

原理

（参考了garfish文档上的介绍）
传统架构：SPA MPA
SPA的DX差（构建时间长 需求相互阻塞 实质上是模块直接耦合程度太高） UX好（体验丝滑 性能好 资源复用）
MPA的DX好（解决了上面的问题） UX差（多应用之间割裂 ）

微前端：取两家之长

  • 子系统间的开发、发布从空间上完成隔离
  • 子系统可以使用不同的技术体系
  • 子系统间可以完成基础库的代码复用
  • 子系统间可以快速完成通信
  • 子系统间需求迭代互不阻塞

整体架构

微前端部署平台: 微前端的服务发现、服务注册、子应用版本控制、多个子应用间同灰度、增量升级子应用、下发子应用信息列表，分析子应用依赖信息提取公共基础库降低不同应用的依赖重复加载

运行时：
为什么不用iframe：

  • 大幅增加资源内存消耗
  • 与上层文档不处于同一个上下文，导致：无法冒泡、难以错误监控、丢失路由状态、无法保持登录态。。。
  • 通信繁琐

基于SPA的架构：
四个核心能力：加载器（loader） 沙箱隔离（sandbox） router store

生命周期：

  1. 渲染
    触发（路由/手动挂载）
    初始化沙箱环境
    判断入口类型（HTML 解析资源 JS 创建挂点DOM）
    副作用交给沙箱处理
    渲染DOM树
  2. 销毁
    销毁应用 销毁副作用 移除DOM

loader的设计：（仅HTML）
下载文件 解析AST 将副作用（style script link等）交给沙箱处理

沙箱的设计：
与组件完全不同的是微前端是由多个独立运行的应用组成的架构风格，这些系统可能分别来自不同的技术体系。项目的开发、测试从空间和时间上都是分离的，由于没有 Iframe 一样原生能力的隔离很难应用间不发生冲突，这些冲突可能会导致应用发生异常、报错、甚至不可用等状态
核心能力：隔离副作用（全局变量、全局事件、定时器、网络请求、localStorage、Style 样式、DOM 元素）
副作用分为静态和动态，静态指HTML标签（style link script） 动态由JS创建
沙箱设计思路：快照模式和VM模式

快照：针对每一种副作用提供一个 Patch 类，这个类需要提供 save 和 load 两个方法。Save 对应着该副作用的环境快照存储，Load 对应着销毁该副作用的销毁恢复环境。 核心思路：存储执行环境=>执行具备副作用的代码=>恢复执行环境
VM模式：快照缺点是在非线性多沙箱实例下不好处理
VM -> 仿虚拟机，提供全新的执行上下文（window document 定时器 localstorage等）

模块联邦

wp5新增
模块联邦可以在多个 webpack 编译产物之间共享模块、依赖、页面甚至应用，通过全局变量的组合，还可以在不同模块之前进行数据的获取，让跨应用间做到模块共享真正的插拔式的便捷使用。比如a应用如果想使用b应用中table的组件，通过模块联邦可以直接在a中进行import(‘b/table’)非常的方便。
分为remote（生产者）和host（消费者），remote暴露模块 host使用

分为本地模块和远程模块
依赖处理：
模块联邦可以将一个应用的包应用于另一个应用，同时具备整体应用(这两个应用)一起打包的公共依赖抽取能力
（可以实现依赖包的自动更新，无需手动）
使用sharedScope（共享作用域实现）

使用：remote引用 exposes暴露模块 shared声明共享模块
(in shared)singleton设置为true时，使用使用sharedScope的共享依赖，当版本不一致时会抛出异常

BFF

随着业务不断的壮大及架构升级，公司不同团队在前后端协同等方面遇到了一些问题：

  • 微服务演进过程中，后端服务/接口骤增，客户端/前端开发需求时，单个页面进行多次请求后端服务，开发效率低下且数据加载时延较高，页面偏卡顿；
  • 客户端/前端的数据模型与服务端的数据模型有一定DIFF，且服务端返回的数据存在冗余；
  • 数据/接口规范前后端可能存在差异，比如字段命名规范、接口数据格式等；
  • 服务端往往在适配客户端/前端不同需求时，需要对数据进行加工，导致服务端掺杂UI逻辑，边界不清晰，不能更好的专注于领域逻辑；
    此时，在前后端增加一层适配层，即BFF，提供接口聚合、字段裁剪、数据规范等，做到前后端解耦，客户端/前端不必受限于服务端，可以更加灵活的调整数据/接口，服务端更专注于自己的领域逻辑。

AGW 以GraphQL作为胶水，业务编写GraphQL查询语句并在AGW平台发布BFF路由，即可完成 HTTP(RestFul) -> GQL -> 后端RPC（也可以是HTTP/TCC等）的流量链路;

场景：多端展示 短生命周期业务 业务聚合

业务部分

闭环开发 开环改造 达人授权码（文件）

  1. 开环代码结构优化（逻辑抽离，之前单页面，现在把请求部分、常量（columns、filter）单独分出，index只复制整体框架）
  2. 公共组件（filterGroup） （高内聚低耦合 可复用可拓展可配置化）
    返回一个Form，根据传来的config渲染对应的form小组件，支持折叠/reset返回预设参数/区分config和groupconfig/对多样的组件进行处理
    auth组件：包装权限访问系统，使用auth相关hook获取用户权限，区分资源类型和authlevel
  3. hook useQuery
    过去直接使用useParams，搜索参数和真正发送给后端的参数不一致，需要进行转换（searchParmas -> queryParams），需要维护多个状态比较混乱
    解决：使用统一的hook处理params，转换后返回query，所有操作都基于query（转换包括多选框选择一个时转换为单数组、时间戳的转换等）
    搜索逻辑：表单改变触发parmasChange，navigate加上参数，使用hook转换后被useeffect检测到，触发请求
  4. 文件上传（可结合分片上传）

解决的问题？
上面的；
做公共组件抽离时没有考虑低耦合，参杂业务逻辑；
重复搜索问题：reset的doreload和params的useeffect叠加，在同表单使用多接口的场景导致异常，解决是不传reset函数（而不是修改组件的reset逻辑）
同表单多接口：请求处加逻辑判断，前端自行分页
